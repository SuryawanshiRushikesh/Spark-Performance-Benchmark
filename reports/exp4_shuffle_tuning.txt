============================================================
EXPERIMENT 4 — Shuffle Partition Tuning
============================================================

Built On:
- Parquet format
- Broadcast joins
- Partition pruning (order_date = 2023-03-15)

Goal:
- Identify optimal shuffle partition count
- Reduce task scheduling overhead
- Improve execution efficiency for pruned workload

Tested Configurations:
- 200 partitions (default)
- 100 partitions
- 50 partitions
- 25 partitions

------------------------------------------------------------
Results
------------------------------------------------------------

Partitions      Time
------------------------------------------------------------
200             11.74s
100              4.49s
50               3.58s
25               3.16s  ← BEST
------------------------------------------------------------

Improvement:
- 25 partitions vs 200 partitions
- ~73% faster execution

------------------------------------------------------------
Observations
------------------------------------------------------------

- 200 shuffle partitions caused excessive scheduling overhead.
- After partition pruning, workload size became small.
- Over-parallelization degraded performance significantly.
- Reducing partitions improved CPU utilization and reduced task overhead.

Conclusion:
Shuffle partition tuning is critical after data reduction.
Optimal partition count depends on workload size.
For this dataset, 25 partitions delivered the best performance.
